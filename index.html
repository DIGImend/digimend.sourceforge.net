DIGImend project

DIGImend — digitizer mending. Improving Linux support for generic graphics tablets.

About

DIGImend project aims at improving Linux support for generic graphics tablets — those which are cheap, widespread, but don't have a proper, dedicated, Linux driver.

Essentially, these are any USB-connected tablets not made by Wacom. For the latter see The Linux Wacom Project.

The project's subjects, instead, are the tablets made by such manufacturers as Genius, Trust, Adesso, DigiPro, Vistablet and others.

Roadmap

The plan, in the first place, is to make these tablets work correctly with the Linux kernel's generic USB HID driver. The project is based on the assumption that the main problem with the kernel support are incorrect or incomplete HID report descriptors which the tablets provide.

All the basic tools needed are completed, but we still need a convenient way for tablet owners to quickly and easily gather tablet information and diagnostics themselves. For this, a diagnostics "wizard" named "digimend-diag" is being implemented.

The next layer to fix is the generic X.org input driver — evdev. It needs to be extended to support tablets better.

At last, the most remote task is to improve user experience by providing GUI tools for tuning tablet response to the input. This they really need — the general quality of such tablets is not very high and input could be quirky and unreliable.

Sub-projects

There is no single program which makes the tablets work, but rather a collection of patches, programs and tools which help make them.

HID report descriptor library and tools — hidrd

Status Release 0.2.0 available
License GPL v2 or later
Sources Release tarballs SVN browse SVN tree
Bugs Browse Submit (please use hidrd category)

About

Hidrd is a library and collection of tools for parsing and generating HID report descriptors. The main documents used during the implementation are Device Class Definition for HID 1.11 and HID Usage Tables 1.12.

Currently, native descriptor format is fully supported. An XML format is developed and supported for input and output. Specification example format is supported for output. Output to C source code with specification example format comments is supported. A tool called "hidrd-convert" is implemented and could be used to convert descriptors between formats.

Roadmap

Some internal redesign should be made, particularly in error reporting. Next, perhaps, specification example input support should be implemented. Then validation needs to be cleaned up (bugs 2950561 and 2941207).

After that, the time may come to implement a stream validation filter with recommendations from HID Parser Error Checking.

HID report descriptor dumping utility — hidrd-dump

Status Deprecated, release 0.1 available
License GPL v2 or later
Sources Release tarballs SVN browse SVN tree
Bugs Browse Submit (please use hidrd-dump category)

About

Hidrd-dump is a tiny utility using libhid to dump HID report descriptors from USB devices. Please note: xxd -r -p chokes on hidrd-dump output (see Debian Bug 567041), so you will need to pass it to tr -d ' \n' first, if you wish to convert it to binary.

Roadmap

Hidrd-dump is replaced by hid-dump — a project based on libusb 1.0.

HID dumping utility — hid-dump

Status Release 1.0 available
License GPL v2 or later
Sources Release tarballs SVN browse SVN tree
Bugs Browse Submit (please use hid-dump category)

About

Hid-dump is a HID dumping utility based on libusb 1.0. It dumps USB HID device report descriptors and reports themselves as they are being sent for all or specific device interfaces.

Roadmap

Hid-dump is quite complete, although a physical descriptor dumping feature may be added in the future.

Usage

Here is the output of hid-dump --help: 

$ hid-dump --help
Usage: hid-dump [OPTION]... <bus> <dev> [if]
Dump a USB device HID report descriptor(s) and/or stream(s).
Arguments:
  bus                      bus number (1-255)
  dev                      device address (1-255)
  if                       interface number (0-254);
                           if omitted, all device HID interfaces
                           are dumped

Options:
  -h, --help               this help message
  -e, --entity=STRING      what to dump: either "descriptor",
                           "stream" or "both"; value can be
                           abbreviated
  -p, --stream-paused      start with the stream dump output paused
  -f, --stream-feedback    enable stream dumping feedback: for every
                           transfer dumped a dot is printed to stderr

Default options: --entity=descriptor

Signals:
  USR1/USR2                pause/resume the stream dump output

You could use hid-dump along with hidrd-convert to dump report descriptors in human-readable format. Like this: 

$ sudo hid-dump --entity=descriptor 2 3 0 | grep -v : | tr -d ' \n' | xxd -r -p | hidrd-convert -o spec
Usage Page (Desktop),               ; Generic desktop controls (01h)
Usage (Mouse),                      ; Mouse (02h, application collection)
Collection (Application),
    Usage (Pointer),                ; Pointer (01h, physical collection)
    Collection (Physical),
        Usage Page (Button),        ; Button (09h)
        Usage Minimum (01h),
        Usage Maximum (03h),
        Logical Minimum (0),
        Logical Maximum (1),
        Report Size (1),
        Report Count (3),
        Input (Variable),
        Report Size (5),
        Report Count (1),
        Input (Constant),
        Usage Page (Desktop),       ; Generic desktop controls (01h)
        Usage (X),                  ; X (30h, dynamic value)
        Usage (Y),                  ; Y (31h, dynamic value)
        Usage (Wheel),              ; Wheel (38h, dynamic value)
        Logical Minimum (-127),
        Logical Maximum (127),
        Report Size (8),
        Report Count (3),
        Input (Variable, Relative),
    End Collection,
End Collection

A tr invocation here is only needed to work around a bug in xxd (see Debian Bug 567041). It is not necessary if it is already fixed in your distribution.

Here is an example of a stream dump from the mouse with report descriptor above: 

$ sudo hid-dump --entity=stream 2 3
000:STREAM                     1280060100.210022
 00 FF 00 00

000:STREAM                     1280060100.217988
 00 FE 00 00

000:STREAM                     1280060100.225985
 00 FC 01 00

000:STREAM                     1280060100.233995
 00 FE 01 00

000:STREAM                     1280060100.241992
 00 FF 01 00

000:STREAM                     1280060100.249995
 00 FE 02 00

000:STREAM                     1280060100.257993
 00 FF 01 00

^C

Here "000" is the interface number, "STREAM" indicates stream chunk and the number to the right is the timestamp in seconds since epoch. The hexadecimal numbers below is the stream itself. Usually every chunk includes a whole report, but if the report is bigger than endpoint's wMaxPacketSize, it will span several chunks.

How to help

Here is what you can do to help the project right now:

  • Dump all your USB input device descriptors using hid-dump and send them in. Any device descriptor will help, but tablets are especially valued, of course.
  • Use project's tools, provide feedback, submit bugs, and feature requests.
  • Contribute your code and join the project, of course!

At last: do anything else you think can help the project, and thank you for your efforts :)!

History

The project founder participated in the development and release of the WizardPen driver 0.5.0 for X.org. After that it became apparent that the method used to make it work was inefficient and quirky. The driver relied on the invalid event stream generated by the kernel USB HID driver, which was confused by invalid HID report descriptors used by some of the supported tablets.

After that, an experiment was made fixing the HID report descriptor of a Genius MousePen 8x6 tablet and patching the kernel to use it instead of the original. The experiment proved successful.

Then, after some attempts, the X.org evdev driver was extended with minimal tablet support. The stack worked perfectly as a whole and it was possible to use the tablet to draw in The GIMP and Inkscape.

It became apparent that in order to fix other devices two things will be needed: a diagnostics collection toolset, the regular users could use to collect information about their tablets, and a tool to efficiently fix their HID report descriptors.

The diagnostics collection wizard was implemented, but it captured generic kernel input stream which could be invalid due to invalid report descriptors, which had a potential of complicating the task. Then, there still was no tool to fix the descriptors. So, it was not released.

The existing "HID Descriptor Tool", provided by the USB Implementers Forum, proved to be insufficient, since it didn't support reading descriptors in their native format and was generally inconvenient and unstable. Thus the HID report descriptor library and tools were started.

DIGImend project

DIGImend — digitizer mending. Improving Linux support for generic graphics tablets.

About

DIGImend project aims at improving Linux support for generic graphics tablets — those which are cheap, widespread, but don't have a proper, dedicated, Linux driver.

Essentially, these are any USB-connected tablets not made by Wacom. For the latter see The Linux Wacom Project.

The project's subjects, instead, are the tablets made by such manufacturers as Genius, Trust, Adesso, DigiPro, Vistablet and others.

Status

Eight tablet models are supported, which are rebranded as more than 30 products. See the compatibility list. Patches are accepted into the kernel and will appear in the 2.6.37 release. Tablet-related fixes are accepted into the evdev X.org driver (xf86-input-evdev) and will appear in the 2.5.99 release. Work has began on adding support for these tablets to xf86-input-wacom driver.

Roadmap

More tablet models are to be supported.

Sub-projects

There is no single program which makes the tablets work, but rather a collection of patches, programs and tools which help make them.

Kernel patches

About

The latest release has support for 8 tablet models, which are rebranded as more than 30 products. Please see the COMPATIBILITY file for details. Only 2.6.32, 2.6.35 and 2.6.36 vanilla kernel patches are available for now.

The patches are accepted into the kernel by Jiri Kosina — the HID core layer maintainer and will appear in the 2.6.37 release.

Dependencies

The tablets require yet unreleased version of the evdev X.org driver (xf86-input-evdev) to work, available from its git repository. Alternatively, the evdev patches could be used to make recently released versions compatible.

Some Waltop tablets are reported to work with the latest versions of the wacom X.org driver (xf86-input-wacom).

Roadmap

Back-porting to older kernel versions used by major distributions will be done on request. Support for other models will be added as they become available to the developer or someone volunteers to aid with diagnostics and testing of their tablets.

The Waltop tablet driver (hid-waltop) will be updated to use "tablet" mode which doesn't cripple the resolution and provides full wheel functions on bigger models.

HID report descriptor library and tools — hidrd

Status Release 0.2.0 available
License GPL v2 or later
Sources Release tarballs GIT browse GIT repository
Bugs Browse Submit (please use hidrd category)

About

Hidrd is a library and collection of tools for parsing and generating HID report descriptors. The main documents used during the implementation are Device Class Definition for HID 1.11 and HID Usage Tables 1.12.

Currently, native descriptor format is fully supported. An XML format is developed and supported for input and output. Specification example format is supported for output. Output to C source code with specification example format comments is supported. A tool called "hidrd-convert" is implemented and could be used to convert descriptors between formats.

Roadmap

Some internal redesign should be made, particularly in error reporting. Next, perhaps, specification example input support should be implemented. Then validation needs to be cleaned up (bugs 2950561 and 2941207).

USB HID dumping utility — usbhid-dump (formerly hid-dump)

Status Release 1.2 available
License GPL v2 or later
Sources Release tarballs GIT browse GIT repository
Bugs Browse Submit (please use usbhid-dump category)

About

Usbhid-dump is a USB HID dumping utility based on libusb 1.0. It dumps USB HID device report descriptors and reports themselves as they are being sent for all or specific device interfaces.

Roadmap

Usbhid-dump is quite complete, although a physical descriptor dumping feature may be added in the future.

Usage

Here is the output of usbhid-dump --help: 

$ usbhid-dump --help
Usage: usbhid-dump [OPTION]...
Dump USB device HID report descriptor(s) and/or stream(s).

Options:
  -h, --help                       output this help message and exit
  -v, --version                    output version information and exit

  -s, -a, --address=bus[:dev]      limit interfaces by bus number
                                   (1-255) and device address (1-255),
                                   decimal; zeroes match any
  -d, -m, --model=vid[:pid]        limit interfaces by vendor and
                                   product IDs (0001-ffff), hexadecimal;
                                   zeroes match any
  -i, --interface=NUMBER           limit interfaces by number (0-254),
                                   decimal; 255 matches any

  -e, --entity=STRING              what to dump: either "descriptor",
                                   "stream" or "all"; value can be
                                   abbreviated

  -t, --stream-timeout=NUMBER      stream interrupt transfer timeout, ms;
                                   zero means infinity
  -p, --stream-paused              start with the stream dump output
                                   paused
  -f, --stream-feedback            enable stream dumping feedback: for
                                   every transfer dumped a dot is
                                   printed to stderr

Default options: --stream-timeout=60000 --entity=descriptor

Signals:
  USR1/USR2                        pause/resume the stream dump output

Warning: please be careful running usbhid-dump as a superuser without limiting your device selection with options. Because usbhid-dump will try to dump every device possible and If you're using a USB keyboard to control your terminal, it will be detached and you will be unable to terminate usbhid-dump and regain control.

If this happens just don't touch your keyboard for the duration of the interrupt transfer timeout (1 minute by default) and the dumping will be aborted.

Here is an example of a report descriptor and stream dump from a mouse: 

$ sudo usbhid-dump --entity=all --address=2:3

 81 06 C0 C0

with 1 minute timeout.

 00 FF 00 00

002:003:000:STREAM             1290272185.217988
 00 FE 00 00

002:003:000:STREAM             1290272185.225985
 00 FC 01 00

002:003:000:STREAM             1290272185.233995

002:003:000:STREAM             1290272185.241992

Here "002" is the bus number, "003" is the device address and "000" is the interface number, "DESCRIPTOR" indicates descriptor chunk and "STREAM" - stream chunk. The number to the right is the timestamp in seconds since epoch. The hexadecimal numbers below is the chunk dumps themselves. Usually every stream chunk includes a whole report, but if the report is bigger than endpoint's wMaxPacketSize, it will span several chunks.

Evdev stream dumping utility — evdev-dump

Status Release 1.0 available
License GPL v2 or later
Sources Release tarballs GIT browse GIT repository
Bugs Browse Submit (please use evdev-dump category)

About

Evdev-dump is a very simple utility for dumping input event device streams.

There exists a similar tool called "evtest". Additionally it can dump device capabilities, but can't dump more than one device at once. It is found in many distributions as a separate package of the same name.

Roadmap

Only bug fixing and occasional minor improvements are expected.

Usage

Here is the output of evdev-dump --help: 

$ evdev-dump --help
Usage: evdev-dump [OPTION]... <device> [device]...
Dump event device(s).

Arguments:
  device           event device path, e.g. /dev/input/event1

Options:
  -h, --help       this help message
  -p, --paused     start with the output paused
  -f, --feedback   enable feedback: for every event dumped
                   a dot is printed to stderr

Signals:
  USR1/USR2        pause/resume the output

How to help

Here is what you can do to help the project right now:

  • Dump all your USB input device descriptors using usbhid-dump and send them in. Any device descriptor will help, but tablets are especially valued, of course.
  • Use project's tools, provide feedback, submit bugs, and feature requests.
  • Contribute your code and join the project, of course!

At last: do anything else you think can help the project, and thank you for your efforts :)!

History

The project founder participated in the development and release of the WizardPen driver 0.5.0 for X.org. After that it became apparent that the method used to make it work was inefficient and quirky. The driver relied on the invalid event stream generated by the kernel USB HID driver, which was confused by invalid HID report descriptors used by some of the supported tablets.

After that, an experiment was made fixing the HID report descriptor of a Genius MousePen 8x6 tablet and patching the kernel to use it instead of the original. The experiment proved successful.

Then, after some attempts, the X.org evdev driver was extended with minimal tablet support. The stack worked perfectly as a whole and it was possible to use the tablet to draw in The GIMP and Inkscape.

It became apparent that in order to fix other devices two things will be needed: a diagnostics collection toolset, the regular users could use to collect information about their tablets, and a tool to efficiently fix their HID report descriptors.

The diagnostics collection wizard was implemented, but it captured generic kernel input stream which could be invalid due to invalid report descriptors, which had a potential of complicating the task. Then, there still was no tool to fix the descriptors. So, it was not released.

The existing "HID Descriptor Tool", provided by the USB Implementers Forum, proved to be insufficient, since it didn't support reading descriptors in their native format and was generally inconvenient and unstable. Thus the HID report descriptor library and tools were started.
